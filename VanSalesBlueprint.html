🎯 SAP B1 (Master System)

  • Master data ownership (Customers, Items, Prices)
  • Final posting system for all transactions
  • Financial accounting & reporting
  • Inventory management
  • DocEntry/DocNum acknowledgment

📱 SalesWorx (Field Execution)

  • Van driver mobile application
  • Sales order creation
  • Invoice generation
  • Collection handling (Cash/iVoucher)
  • Returns processing
  • Journey management

🔄 Middleware (Bridge)

  • MS SQL Server database
  • Bidirectional data sync
  • Outbound: Masters, Stock, Open Invoices
  • Inbound: Customers, Invoices, Returns, Collections
  • Data transformation layer

📦 Inventory Model

  • Non-inventory items (UDT-based stock tracking)
  • Van = Virtual Warehouse
  • Stock tracked via User Defined Tables
  • No standard OITW warehouse movement
  • Opening/Closing stock reconciliation

Acknowledgment Flow

Document Posting Acknowledgment

  1. SalesWorx sends transaction to Middleware staging table
  2. Inbound service picks up pending records
  3. Service posts to SAP via DI API
  4. SAP returns DocEntry/DocNum on success
  5. Service updates staging table with SAP DocEntry
  6. SalesWorx queries acknowledgment from middleware
  7. Transaction marked as complete in SalesWorx
-- Acknowledgment Query SELECT SWX_InvoiceID, SAP_DocEntry, SAP_DocNum, Status, ProcessedAt FROM SWX_Invoice_Staging WHERE Status = 'Posted' AND SWX_InvoiceID = @SWX_InvoiceID

Problem Scenarios

🔴 Duplicate Customers

Scenario: Same retailer created multiple times with different CardCodes due to driver creating "new" customer instead of selecting existing.

Impact: Split receivables, incorrect customer history, reporting issues.

Root Cause: No duplicate check before customer creation, poor search in mobile app.

CardCodeCardNamePhoneBalance
C-00125Al Noor Grocery+9665012345675,200
C-00289Alnoor Grocery+9665012345672,100
C-00412Al-Noor Grocery Shop+966-50-12345671,800

Resolution: Merge customers in SAP, update all invoices/payments to master CardCode, implement phone-based duplicate check in middleware.

🔴 Tender Mismatch

Scenario: Collection recorded as Cash in SalesWorx but iVoucher payment was actually received.

Impact: Cash reconciliation fails, incorrect GL postings, audit findings.

Root Cause: Driver selected wrong tender type, no validation at point of collection.

Posted (Wrong)
DR: Cash in Hand 1,150
CR: Accounts Receivable 1,150
Should Be (Correct)
DR: iVoucher Clearing 1,150
CR: Accounts Receivable 1,150

Resolution: Cancel wrong payment, repost with correct tender. Implement iVoucher QR scan validation.

🔴 Cross-Day Closing Issues

Scenario: Driver closes journey at 12:30 AM but transactions dated previous day. Business date vs system date mismatch.

Impact: Transactions posted to wrong period, daily reconciliation fails, reporting gaps.

Root Cause: No business date enforcement, journey not closed before midnight.

InvoiceBusiness DateSystem DateClose TimeStatus
INV-0012024-03-152024-03-1522:30OK
INV-0022024-03-152024-03-1600:30MISMATCH
INV-0032024-03-152024-03-1600:45MISMATCH

Resolution: Use Business Date from journey, not system date. Force journey close before midnight cutoff.

🔴 Stock Variance at Journey End

Scenario: Physical count doesn't match system stock. Opening + Loads - Sales - Returns ≠ Closing.

Impact: Inventory discrepancy, potential theft, P&L impact.

Root Cause: Unrecorded damages, theft, transaction not synced, counting errors.

ItemOpeningLoadedSoldReturnedExpectedActualVariance
SKU-00150100120535350
SKU-00230506002017-3
SKU-00325758022224+2

Resolution: Document variance, create adjustment entry, investigate root cause. Implement damage recording in app.

🔴 Invoice Posting Failure

Scenario: Invoice stuck in staging, not posted to SAP due to validation error.

Impact: Revenue not recognized, customer not billed, stock discrepancy.

Root Cause: Invalid ItemCode, inactive customer, price list mismatch, missing tax code.
